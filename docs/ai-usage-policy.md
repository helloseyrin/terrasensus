# TerraSensus — AI Usage Policy

*This document defines where AI-generated content is and is not acceptable in TerraSensus.*
*It must be read by every contributor before modifying any AI-related code, prompt, or threshold.*
*Last updated: 2026-03-16*

---

## Premise

TerraSensus works with **real crops, real money, and real food**. A Ukrainian watermelon farm
running this app may be making irrigation decisions that affect several tonnes of food.
A Central Asian cotton farmer acting on a bad EC recommendation could cause irreversible
salt damage to their soil — damage that takes years to reverse, if it reverses at all.
A vineyard owner following wrong pH advice could lose an entire vintage.

This is not a productivity app. Errors have physical consequences.

---

## Where AI MUST NOT Be Used

### 1. Critical and warning threshold alerts
**Who does it instead**: `services/alert-engine/rules.py` — synchronous, local, no network call.

The alert engine evaluates sensor readings against crop-aware thresholds with zero
external dependencies. It runs whether or not Gemini is up, whether or not the farmer
has internet, whether or not Anthropic's API is reachable.

A "critical low moisture" alert for Mykola's watermelons must fire the moment the sensor
reading crosses the threshold. Not after a 3-second AI API call. Not after a Pub/Sub
message round-trip. Immediately, locally, deterministically.

**Rule**: No AI API call may sit in the path between a sensor reading and a critical alert.

### 2. Drought risk scoring
**Who does it instead**: `services/alert-engine/drought.py` — rule-based scoring from
Open-Meteo rainfall + moisture trend.

Drought risk is time-sensitive. A farmer who receives a drought warning 4 hours late
because an AI API was slow may have already missed their irrigation window. Rule-based
scoring is instantaneous.

### 3. Any calculation involving farmer financial records
**Who does it instead**: deterministic SQL/Python in the ingestion or analytics service.

ROI calculations, fertiliser spend totals, "unnecessary application" flags — these
are financial records. A farmer may use them to claim subsidies, dispute invoices,
or make investment decisions. They must be reproducible, auditable, and identical
every time they are computed. AI is non-deterministic. It must not touch financial
calculation logic.

### 4. Sensor reading storage, validation, or routing
**Who does it instead**: ingestion service + Pub/Sub pipeline.

Sensor readings go from the simulator (or real hardware) through Pub/Sub to the
ingestion service to Cloud SQL. No AI sits in this pipeline. Data flows regardless
of model availability.

### 5. Decisions described as certain or prescriptive to the farmer
AI outputs are **suggestions**, not **instructions**. The UI must never display an
AI recommendation with language like "you must apply" or "do not irrigate". Always:
"Based on your current readings, we suggest..." with the agronomist disclaimer visible.

## Where AI IS Used (with conditions)

### ✓ Plot onboarding education
**When**: immediately after a farmer registers a plot.
**What**: soil portrait, crop-specific threshold explanations, regional risk summary,
one concept to learn, regenerative note.
**Why it's acceptable**: educational content, not operational instruction. The farmer
is not about to act on this — they are learning context. A wrong fact here is
correctable. A wrong threshold alert at 3am is not.
**Condition**: clearly labelled as AI-generated. Agronomist disclaimer visible.
Cached for 30 days — does not re-call the API on every visit.

### ✓ Crop and fertiliser recommendations (on-demand)
**When**: farmer explicitly requests a recommendation from the recommendations screen.
**What**: soil health assessment, corrective actions, crop suitability, fertiliser type
and timing suggestions.
**Why it's acceptable**: on-demand means the farmer chose to ask. They are not acting
on an unsolicited push notification. They can read, consider, and decide.
**Conditions**:
- Multi-model fallback chain — never blocked by single provider outage
- Agronomic bounds checker — numeric values validated before display
- Agronomist disclaimer on every response
- Model source shown ("Generated by Gemini" / "Generated by Claude" / "Guideline-based")
- One-tap "flag this recommendation" button always visible

### ✓ Ecological scoring and regenerative alternatives
**When**: alongside any fertiliser recommendation.
**What**: eco-score (A–F), GHG contribution estimate, biodiversity note, alternative
approaches (cover crops, biochar, mycorrhizal inoculants, integrated pest management).
**Why it's acceptable**: informational, not operational. A wrong eco-score does not
cause crop damage.
**Conditions**: same as crop recommendations above.

### ✓ Lab report parsing (Document AI + Gemini Vision)
**When**: triggered by a lab report file upload.
**What**: extracts structured N-P-K-pH-EC-OM values from unstructured PDF or image.
**Why it's acceptable**: document extraction, not agronomic reasoning. The AI is
doing OCR + structure extraction, not giving advice.
**Conditions**:
- Every extracted value displayed to the farmer for review before being stored
- "Please verify these values match your original lab report" prompt on every parse
- If extraction confidence is low (Gemini Vision fallback path): flag entire report
  for manual entry rather than storing potentially wrong values

### ✓ Supplier recommendations
**When**: alongside fertiliser recommendations, on-demand.
**What**: product names, application rates, estimated costs, eco-scores, local supplier
suggestions.
**Why it's acceptable**: purchasing decisions are not irreversible in the same way crop
decisions are. A farmer can choose not to buy a recommended product.
**Conditions**: same as crop recommendations. Flag if recommended application rate
exceeds agronomic bounds.

## Conditions That Apply to ALL AI Usage

1. **Multi-model fallback chain** — Gemini → Claude → rule-based local. Always.
2. **Agronomic bounds checker** — all numeric values validated before display.
3. **Agronomist disclaimer** — visible on every AI-generated screen.
4. **Model source label** — farmer knows whether they are reading AI output or a guideline.
5. **Flag button** — one-tap access to report a bad recommendation (creates GitHub `ai-quality` issue).
6. **Cached responses** — never re-call the API for the same content unnecessarily. Serve cached responses during outages with a freshness label.
7. **⚠ UNVALIDATED label** — all threshold values and prompt outputs are flagged until reviewed by a domain expert agronomist. This label is not removed by code review. It is removed by agronomist sign-off.

## The Responsibility Boundary

TerraSensus is a **decision support tool**, not a **decision maker**.

| TerraSensus responsibility | Farmer responsibility |
|---|---|
| Provide accurate sensor data | Verify sensors are calibrated and functioning |
| Fire timely threshold alerts | Act on alerts |
| Provide AI-generated suggestions | Evaluate suggestions against their own knowledge |
| Label AI uncertainty honestly | Consult a qualified agronomist for high-stakes decisions |
| Validate AI outputs against agronomic bounds | Not rely solely on TerraSensus for food-security decisions |

This boundary is not a liability disclaimer. It is an honest description of what the
software knows and does not know. A software engineer built this framework.
Agronomists, farmers, and regional experts must validate it before it is trusted with
food security.

---

## The Validation Checklist (Before Removing ⚠ UNVALIDATED Flags)

No threshold value, AI prompt, or recommendation template should be treated as
agronomically authoritative until the following are done:

- [ ] A qualified agronomist has reviewed `services/alert-engine/rules.py` CROP_THRESHOLDS
- [ ] Onboarding prompt output has been tested against real plots for each crop type
- [ ] Recommendation prompt output has been evaluated against at least 10 real soil profiles per crop
- [ ] Regional risk outputs have been reviewed by someone with regional agricultural knowledge
  (Ukrainian watermelon agronomy for Kherson, Central Asian agronomy for cotton, Pacific Northwest viticulture for Pinot Noir)
- [ ] Agronomic bounds in `client.py: AGRONOMIC_BOUNDS` have been confirmed as safe maximums
- [ ] At least one farmer from each persona region has reviewed the onboarding output for their crop

Until this checklist is complete, TerraSensus is a **demonstration framework**, not a
**production agricultural advisory tool**.
