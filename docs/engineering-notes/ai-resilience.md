# Engineering Notes — AI Resilience for Food-Security Contexts

This documents the reasoning behind the multi-model fallback chain and why standard
"use one provider" AI integration is insufficient for TerraSensus.

---

## Why Single-Provider AI is Not Acceptable Here

TerraSensus may be used by farmers in Kherson Oblast (Ukraine), the Ferghana Valley
(Uzbekistan), or other regions where soil management decisions directly affect food
supply. A recommendation missed due to a provider outage, or acted on because it was
wrong, has real-world consequences.

The standard SaaS approach — "call OpenAI / Gemini, handle errors gracefully" — is
not adequate here. Graceful degradation must mean something more than an error message.

---

## Two Separate Failure Modes

### 1. Provider outage (availability failure)
The API returns an error, times out, or is unreachable.

**Mitigation**: fallback chain → Gemini → Claude → rule-based local knowledge.
Gemini and Claude run on entirely separate infrastructure (Google vs Amazon). A GCP
regional incident will not affect Anthropic's API.

### 2. Model incorrectness (correctness failure)
The API returns a successful response with agronomically wrong advice. This is harder
to detect than an outage — the response is well-formatted, confident, and wrong.

**Mitigation**: agronomic bounds checker (`client.py: check_bounds()`). Every numeric
value in a recommendation is validated against hard limits before it reaches the farmer.
Values outside bounds trigger a fallback to the next model, not just a warning.

Limits are intentionally conservative and flagged `⚠ UNVALIDATED` pending agronomist
review. Conservative + unvalidated is safer than precise + unvalidated.

---

## What the Fallback Chain Guarantees

| Scenario | What the farmer sees |
|---|---|
| Gemini up, bounds pass | Full AI recommendation, labelled "Generated by Gemini" |
| Gemini down, Claude up | Full AI recommendation, labelled "Generated by Claude" |
| Both down, internet up | Guideline-based response from local knowledge, clearly labelled |
| No internet | Cached last recommendation (mobile AsyncStorage) with "last updated X ago" |
| All fail | The rule-based alert engine still fires critical alerts — the safety net never calls an AI |

The farmer always receives *something*. Critical alerts (moisture, EC, pH thresholds)
are never blocked by AI availability — they run entirely in `rules.py` with no network
dependency.

---

## The Disclaimer is Not Optional

Every AI response displayed in the app carries:

> "This recommendation is AI-generated guidance. For decisions affecting food security
> or significant financial investment, consult a qualified agronomist."

This is not legal boilerplate. It is accurate. The thresholds in `rules.py` are flagged
unvalidated. The AI has not been evaluated by a domain expert for Central Asian cotton
agronomy or Kherson watermelon cultivation. Until that validation happens, this disclaimer is
the honest truth about what TerraSensus can and cannot claim.

---

## The Bounds Checker is Not a Substitute for Expert Review

The bounds checker catches dangerous numeric values (e.g. "apply 500 kg/ha N").
It cannot catch:
- Subtly wrong qualitative advice ("cotton prefers waterlogged soil" — false)
- Region-specific errors ("apply more potassium" — correct globally, wrong on K-rich chernozem)
- Timing errors ("apply nitrogen now" — correct for spring wheat, wrong if frost is forecast)

These require agronomic domain expert review of the prompts and evaluation of real
model outputs against known-good scenarios. This is a prerequisite for removing the
`⚠ UNVALIDATED` flags from `CROP_THRESHOLDS`.

---

## Response Caching Strategy

Onboarding responses are expensive (multi-paragraph AI generation) and change slowly
(the soil portrait for a Kherson watermelon field doesn't change week to week). Cache them:

- **Server**: `onboarding_cache` table in Cloud SQL, TTL 30 days, invalidated if crop or
  region changes
- **Client**: React Native AsyncStorage, no TTL — serves even when offline
- **Freshness label**: mobile UI shows "Last updated X days ago" on cached responses

Full recommendations and supplier suggestions are cached with a 24-hour TTL — soil
conditions can change meaningfully in 24 hours (rain event, irrigation).

---

## Logging AI failures as engineering incidents

Every fallback event is logged with:
- Timestamp
- Which model failed
- Error message
- Call type (onboarding / recommend / supplier)
- Plot ID and crop (no farmer PII)

These logs feed `docs/lessons-learned.md` and the `ai-quality` GitHub issue template.
Pattern: if Gemini fails more than 3 times in 24 hours for the same call type, create
an issue automatically. This is not implemented yet — flag for v2.
